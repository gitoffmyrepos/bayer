Architecture Overview
Five-Layer Integration Architecture

The solution follows a medallion-style architecture that progressively refines data as it moves from source systems through AWS into Model N. Each layer has a distinct responsibility, ensuring clean separation of concerns, auditability, and recoverability at every stage of the pipeline.

1

Acquisition / Consumption Layer

External SAP systems and partner integrations that produce the raw enterprise data -- procurement, supply chain, master data, HR, payments, and third-party files via Axway.

2

Landing & Raw Layer

First point of receipt in AWS. Files land in S3 in their original format, then are cataloged and archived in an immutable raw zone that serves as the single source of truth for reprocessing.

3

Enriched / Stage Layer

Business logic, data quality checks, and transformations via Glue, Lambda, Athena, and Snowflake. Data here has been validated, standardized, and is ready for downstream consumption.

4

Model N Application Layer

Model N PhyN Full Suite (SaaS) -- the core engine for government pricing, commercial contracting, Medicaid rebates, chargeback management, and 340B compliance calculations.

5

Reporting & Analytics Layer

Galaxy Reports, Model N UI, and DBX-FI deliver revenue analytics, contract performance dashboards, pricing compliance, and financial reconciliation reports to business stakeholders.

All layers are underpinned by Common Platform Services providing governance, security, monitoring, DevOps, and infrastructure-as-code capabilities.

Layer 2a
S3 Landing Zone — First Receipt

The Landing Zone is the first point of contact for all incoming data within the AWS platform. Files from SAP systems, Axway, and SFTP connections are deposited here in their original format -- CSV, XML, Parquet, or JSON. Data is organized by source system, date, and batch identifier, ensuring complete traceability from the moment it enters the platform.

☁ Amazon S3 (Landing Zone)
Object storage serving as the initial deposit point. Every file is tagged with metadata including source system, timestamp, batch ID, and file checksum. This ensures that no data is lost or duplicated during ingestion.
CSV XML Parquet JSON
🔌 Ingestion Patterns

Bulk / Historical Load: Used during initial migration and periodic full refreshes. Transfers multi-year transaction histories (millions of records) into the landing zone during go-live cutover.

Incremental Load: Day-to-day operations capturing only new or changed records. Reduces processing time, lowers costs, and ensures near-real-time data availability.

SFTP: Secure file transfers for flat file exchanges with Axway and external partners delivering CSV, XML, and fixed-width files.

Key Principle: Data lands as-is. No transformations occur here. This preserves the original source format for audit trails and reprocessing scenarios.
Layer 2b
S3 Raw Zone — Immutable Archive

After initial cataloging and partitioning, data is copied from the Landing Zone to the Raw Zone. This is the immutable historical record -- data here is never modified or deleted during normal operations. It serves as the single source of truth and the foundation for any reprocessing, audit investigation, or disaster recovery scenario.

🗃 Immutable Storage

The Raw Zone stores a copy of the landing data after AWS Glue crawlers catalog the data structure through schema discovery. Data is partitioned by source, date, and entity type for efficient querying.

This zone enables full reprocessing of the pipeline from any historical point without returning to source systems -- critical for regulatory compliance and SOX audit requirements.

🛠 AWS Glue Processing

Glue Crawlers: Automatically discover data schemas, detect column types, and register tables in the Glue Data Catalog for downstream SQL access via Athena.

Glue ETL Jobs: Handle format conversion and partitioning. Convert raw CSV/XML into optimized columnar formats (Parquet) for faster, cheaper querying in the enriched layer.

AWS Glue Crawlers Glue Data Catalog Amazon S3 Parquet AWS Glacier (archival)
Retention: Historical data is retained for compliance, then transitioned through S3 Standard → S3 Infrequent Access → Glacier via custom lifecycle management services, reducing storage costs by up to 80%.
Layer 3
ETL Enrichment — The Transformation Engine

The Enriched Layer applies business logic, validates data quality, and prepares data for consumption by Model N and reporting systems. This is where raw data becomes business-ready through a combination of AWS Glue, Lambda, Athena, Snowflake, and a dedicated Data Quality Control framework.

⚙ AWS Glue ETL
Executes transformation jobs applying business rules, joins datasets across sources, standardizes formats, and writes enriched outputs to the Enriched S3 Zone.
⚡ AWS Lambda
Event-driven serverless compute for lightweight tasks: file validation (headers, row counts, completeness), triggering downstream workflows, and custom transformation logic.
🔍 Amazon Athena
Serverless SQL engine querying S3 data directly. Used for data validation checks, ad-hoc exploration during development, and supporting data governance audits.
❄ Snowflake
Cloud data warehouse for complex transformations, cross-source joins, and analytics workloads that benefit from Snowflake's performance optimization engine.
📋 File Validation
Validates incoming files against expected schemas -- column names, data types, row counts, file sizes, mandatory fields. Invalid files are quarantined and flagged.
✅ DQC Framework
Dedicated data quality control: completeness, accuracy (expected ranges), consistency (cross-source agreement), and timeliness (arrival schedule adherence).
Output: Validated, standardized, enriched datasets in S3 Enriched Zone + Snowflake -- ready for Model N ingestion and reporting consumption.
Orchestration
Pipeline Orchestration — Step Functions & CloudWatch

The end-to-end data pipeline is coordinated by AWS Step Functions, which define state machines governing every processing step from ingestion through transformation to delivery. Each data pipeline -- such as "SAP P2R daily load" -- is expressed as its own Step Function workflow with built-in error handling, retries, and parallel execution paths.

🎯 AWS Step Functions

Primary orchestration engine defining state machines for the complete data processing sequence. Each state machine coordinates:

  • Sequential and parallel processing steps
  • Automatic retries with exponential backoff
  • Error branching and dead-letter handling
  • Approval gates for sensitive operations
  • Cross-pipeline dependency management
📈 Amazon CloudWatch
Monitors pipeline execution in real time. Triggers alarms when jobs fail, run longer than expected, or produce unexpected output volumes. CloudWatch Events schedule Step Functions (e.g., daily at 02:00 UTC) and provide centralized logging across all services.
🔌 AWS Lambda (Glue Logic)
Acts as the connective tissue between orchestration steps -- invoked by Step Functions to perform file validation, send notifications via SNS, update status tracking, and trigger downstream workflows.
🆕 ServiceNow Integration
Pipeline failures and infrastructure incidents automatically create ServiceNow tickets. Change requests follow the ServiceNow approval workflow for infrastructure modifications.

Common Platform Services
Security, Governance & Compliance

A comprehensive shared services layer supports all architecture layers with cross-cutting security, governance, and compliance capabilities. Every API call is logged, every data access is controlled, and every resource is provisioned through infrastructure-as-code -- ensuring full auditability for SOX compliance and regulatory requirements.

🔒 Security

AWS IAM: Controls who and what can access AWS resources. Defines roles, policies, and permissions for users, service accounts, and workloads, enforcing least-privilege access.

AWS KMS: Manages encryption keys protecting data at rest (S3, DynamoDB, Snowflake) and in transit. All platform data is encrypted using KMS-managed keys.

🕵 Audit & Compliance

CloudTrail: Records every API call -- who did what, when, and from where. Custom extensions enrich logs with business context, linking API calls to specific pipeline runs. Essential for SOX compliance.

CloudWatch: Collects logs, metrics, and traces from all components. Custom dashboards provide real-time visibility into pipeline health, error rates, and processing times.

📚 Data Governance

Amazon Athena: SQL-based data exploration for governance queries. Data stewards audit lineage, run quality checks, and investigate anomalies across the S3 data lake.

AWS Lake Formation: Fine-grained access control over databases, tables, and columns. Enforces data classification policies and tracks access for compliance.

🛠 DevOps & CI/CD

GitHub + Terraform: Every AWS resource is defined in Terraform IaC. Pull requests enforce code review. Changes deploy through GitHub Actions CI/CD pipelines.

Amazon SNS: Notifications for pipeline events -- completions, failures, approvals -- delivered via email, Slack, or PagerDuty to the operations team.

IAM KMS Lake Formation CloudTrail CloudWatch Terraform GitHub Actions SNS

AWS Lambda Functions — Complete Inventory

Eight Lambda functions power the event-driven backbone of the platform — handling SFTP file triggers, Step Function orchestration, audit logging, error notifications, web scraping, and credential management.

⚡ bay-modeln-sftp-file-stepfn-trigger
Listens for file arrivals on the SFTP server (S3 events). Triggers the inbound Step Function workflow when new data files land. The entry point for all file-based data flows.
⚠ Legacy User/Password Auth — Snowflake insert
📋 bay-modeln-initialize-parameter-stpfnc
Initializes Step Function parameters at workflow start. Retrieves processing flags (Raw, Enriched, Model, RODB) from DynamoDB config tables to determine which Glue jobs to execute.
📥 bay-modeln-fetch-source-file-to-S3
Fetches source files from the SFTP server and deposits them in the S3 Landing Zone. Handles file validation, generates flag files for downstream processing, and archives existing files.
📜 bay-modeln-capture-audit-info
Captures audit information at START and END of every pipeline run. Updates DynamoDB audit tables with execution status, timestamps, and failure details for compliance tracking.
📨 bay-modeln-email-attachment-error
Sends email notifications with job status and error attachments when pipeline failures occur. Ensures the operations team is immediately alerted to data processing issues.
⚠ Legacy User/Password Auth
🔎 bay-modeln-scraping-Lambda
Automates web scraping for Treasury Bill (TBILL) and Consumer Price Index (CPI-U) data. Runs weekly/monthly on schedule and triggers Step Functions for downstream processing.
🔑 bay-modeln-rodb-creds-update-lambda
Manages Oracle RODB credential rotation. Updates connection credentials in Secrets Manager, includes Oracle client library layer for secure database connectivity.
👥 bay-modeln-sftp-user-provisioner
Synchronizes Entra ID (Azure AD) groups to SFTP user accounts. Runs daily at 5:30 PM UTC. Auto-provisions and de-provisions users, manages IAM roles and SSH keys.
ETL Processing
AWS Glue Jobs — Complete Inventory

Six production Glue jobs form the ETL backbone, processing data through the Raw, Enriched, Model, and RODB layers. Each runs on Apache Spark (Glue 4.0) or Python Shell, connected to Snowflake, Oracle RODB, and SFTP endpoints via VPC.

📦 bay-modeln-raw-job
Spark 4.0 G.1X × 10
Processes files from the Landing Zone by fileGroupID. Applies file-level validation rules, partitions data, and moves valid records to the Raw Zone. Rejects go to quarantine for investigation.
⚙ bay-modeln-model-job
Spark 4.0 Snowflake VPC
Retrieves Snowflake stored procedure names from DynamoDB config, then executes analytical ETL procedures in Snowflake. The bridge between AWS data lake and Snowflake analytics.
📚 bay-modeln-rodb-job
Spark 4.0 Oracle JDBC VPC
Connects to Model N's Oracle Read-Only Database (RODB) via JDBC. Fetches flex fields and domain-specific data, writes CSV output to S3 Raw Zone, then triggers Snowflake procedures for enrichment.
🔬 bay-modeln-enriched-job
Spark 4.0 XML/CSV
Reads from the Raw + processed data, applies business transformation rules, and creates output XML/CSV files in the Enriched Zone outbound path — ready for delivery to Model N or partner systems.
🚀 bay-modeln-s3-to-sftp-pythonshell-job
Python Shell 3.9 Paramiko/SFTP VPC
Pulls processed files from the S3 Enriched outbound path and delivers them to target SFTP servers (Model N, Axway, Galaxy, SAP CPI). Archives files after successful transfer with load_id and UTC timestamps.
🛰 Glue VPC Connection
NETWORK us-east-1b
Shared VPC network connection used by Model, RODB, and S3-to-SFTP jobs. Enables Glue workers to reach Snowflake, Oracle RODB, and SFTP endpoints within the private network — no public internet traversal.
Data Warehouse Integration
Snowflake Connectivity — How AWS Connects to Snowflake

Snowflake serves as the central analytics warehouse. Data flows through a secure, private connection using IAM federation for S3 access, key-pair authentication for direct queries, and Spark connectors for ETL processing.

🔑 Authentication Methods
Primary: RSA Key-Pair Auth
PEM-formatted RSA private keys stored in AWS Secrets Manager. Converted to DER/PKCS8 format for Snowflake connector. No passwords in transit.
Legacy: User/Password
Still used in some older Lambda functions. Being migrated to key-pair auth across all services.
🔒 IAM Federation for S3 Access
IAM Role: bay-modeln-s3-sf-int-us-east-1
